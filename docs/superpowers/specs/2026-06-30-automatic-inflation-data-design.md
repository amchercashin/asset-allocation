# Автоматическое обновление данных по инфляции

Дата: 2026-06-30  
Статус: утверждено пользователем

## Цель

Автоматически поддерживать график ИПЦ на сайте `asset-allocation` в актуальном состоянии, используя официальный источник, не меняя уже опубликованную историю и не делая работоспособность страницы зависимой от доступности внешнего сервиса в браузере посетителя.

Готовое решение должно:

- получать месячный ИПЦ по Российской Федерации из официального источника;
- сохранять прежние даты и значения ряда без изменений;
- добавлять только новые месяцы;
- прекращать обновление при изменении истории, пропуске месяца, ошибке структуры или недоступности источника;
- оставлять последнюю корректную версию сайта доступной при любой ошибке обновления;
- обновлять и данные в Git, и опубликованный GitHub Pages-сайт;
- поддерживать ручной запуск процедуры;
- показывать на странице, что данные обновляются автоматически из Росстата.

## Исходное состояние

- Сайт публикуется из корня ветки `master` через GitHub Pages в режиме `legacy`.
- Рабочая основа — `origin/master` на коммите `87eada7`.
- ИПЦ находится в `ipc_data.js` как вручную внесённая таблица месячных процентов с вычислением накопленного индекса в браузере.
- Ряд нормирован к февралю 2003 года (`1.0`), использует последние календарные дни месяцев и заканчивается июнем 2025 года.
- Значения официального ряда Росстата до мая 2020 года совпадают со старым жёстко заданным рядом с максимальным отклонением менее `5e-10`; расхождение связано только с округлением.
- В основной локальной папке есть отдельная незакоммиченная переработка `src/`. Она не входит в эту задачу и остаётся нетронутой; работа выполняется в изолированном worktree от `origin/master`.

## Источник

Первичный источник — раздел Росстата «Цены, инфляция»:

<https://rosstat.gov.ru/statistics/price>

На странице публикуется актуальная таблица с именем вида `ipc_mes_MM-YYYY.xlsx`. В таблице лист `01` содержит месячные индексы потребительских цен по Российской Федерации, выраженные в процентах к концу предыдущего месяца. Таблица содержит полную историю, поэтому новое состояние можно проверить независимо от локального файла.

Жёстко фиксировать URL конкретного XLSX нельзя: имя меняется при каждой публикации, а прежний стабильный псевдоним `ipc_mes.xlsx` уже возвращает `404`. Обновитель каждый раз читает официальную страницу, выбирает самый свежий подходящий XLSX на домене Росстата и сохраняет фактический URL и SHA-256 файла в метаданных.

ЕМИСС/Fedstat не используется как основной источник: сайт блокирует обычные автоматические запросы (`403`) и не предоставляет столь же простой устойчивый публичный интерфейс. StatBureau не является первичным государственным источником. Банк России публикует годовую инфляцию и аналитические материалы, но не заменяет полный месячный ряд Росстата.

## Выбранная архитектура

Обновление выполняется на GitHub Actions, а не в браузере.

Причины:

- GitHub Pages — статический хостинг без серверного Python;
- XLSX Росстата не отдаёт CORS-заголовок для клиентского чтения;
- имя файла меняется;
- клиентский парсер XLSX увеличил бы размер и сложность страницы;
- при клиентском варианте отказ Росстата ломал бы график каждому посетителю;
- статический сгенерированный файл продолжает работать независимо от внешних сервисов.

Компоненты:

1. `data/inflation-monthly.json` — канонический накопительный файл с месячными индексами и метаданными источника.
2. `scripts/update_inflation.py` — загрузка страницы и XLSX, разбор, проверки, добавление новых месяцев и генерация `ipc_data.js`.
3. `ipc_data.js` — совместимый с существующим сайтом статический артефакт. Интерфейс глобальной переменной `ipc` (`x`, `y`, `name`, `type`, `line`) сохраняется.
4. `tests/` — тесты парсера, правил append-only, генерации, совместимости истории и ошибок.
5. `.github/workflows/pages.yml` — проверки, обновление данных, commit и публикация Pages.

## Поток данных

1. Workflow запускается еженедельно в непиковую минуту, вручную или при push в `master`.
2. Для расписания и ручного запуска обновитель читает страницу Росстата.
3. Из ссылок выбирается XLSX с максимальным периодом `MM-YYYY` и только с `https://rosstat.gov.ru/`.
4. Файл загружается и открывается через `openpyxl`.
5. На листе `01` по содержимому, а не по жёстким номерам ячеек, находятся строка годов и 12 строк месяцев.
6. Полученный официальный ряд сравнивается со всем сохранённым пересечением.
7. Если пересечение идентично, добавляются новые последовательные месяцы.
8. Из канонического файла заново генерируется `ipc_data.js` с округлением до 10 знаков, как в опубликованной версии.
9. Выполняются все тесты и синтаксическая проверка JavaScript.
10. При изменении данных workflow делает commit вида `data: update CPI through YYYY-MM` и push в `master`.
11. Независимо от того, создался ли commit, успешный workflow собирает статический artifact и публикует его через стандартный GitHub Pages deployment.

Commit, сделанный штатным `GITHUB_TOKEN`, не запускает новый Pages build. Поэтому обновление и deployment находятся в одном workflow. Режим Pages один раз переключается с `legacy` на GitHub Actions.

Документация GitHub:

- <https://docs.github.com/en/pages/getting-started-with-github-pages/creating-a-github-pages-site>
- <https://docs.github.com/en/pages/getting-started-with-github-pages/using-custom-workflows-with-github-pages>
- <https://docs.github.com/en/actions/reference/workflows-and-actions/events-that-trigger-workflows#schedule>

## Формат канонических данных

`data/inflation-monthly.json` содержит:

- `schema_version`;
- `source.agency`, `source.landing_url`, `source.file_url`;
- `source.retrieved_at`, `source.file_sha256`;
- `latest_period`;
- `months`: упорядоченный массив объектов `period` (`YYYY-MM`) и `index_to_previous_month`.

В Git не сохраняется загруженный XLSX: он нужен только как вход проверки. JSON остаётся обозримым, версионируемым и пригодным для восстановления генерации без сети.

## Инварианты и обработка ошибок

Обновление завершается ошибкой до commit и deployment, если выполняется хотя бы одно условие:

- официальная страница или XLSX недоступны;
- ссылка ведёт не на разрешённый HTTPS-домен Росстата;
- лист или заголовки таблицы не найдены;
- год или месяц неоднозначны;
- в диапазоне от февраля 2003 года есть пропуск или дубликат;
- значение не является числом или выходит за защитный диапазон `80..120`;
- любое ранее сохранённое месячное значение изменилось;
- официальный файл оказался старше локального ряда;
- результат не воспроизводит зафиксированный baseline `ipc_data.js` до июня 2025 года.

При ошибке workflow становится красным и отправляет стандартное уведомление GitHub владельцу. Никакие файлы и Pages deployment не меняются, поэтому посетитель продолжает получать последнюю корректную версию.

Исторические пересмотры Росстата не применяются автоматически. Они требуют ручного решения и отдельного изменения baseline: это сознательная защита от незаметной порчи графика.

## Расписание и эксплуатация

- Плановый запуск: один раз в неделю, чтобы не зависеть от точного дня публикации Росстата.
- Минута запуска не равна `00`, поскольку GitHub предупреждает о задержках и пропуске scheduled jobs в начале часа.
- `workflow_dispatch` позволяет запустить обновление вручную.
- В публичных репозиториях GitHub отключает schedule после 60 дней без активности. При нормальной работе ежемесячный commit данных поддерживает активность; ручной запуск остаётся аварийным способом восстановления.
- Workflow получает минимальные права: `contents: write`, `pages: write`, `id-token: write`; остальные права не выдаются.
- Одновременные Pages-запуски объединяются через `concurrency`, чтобы обновления не конкурировали.

## Изменения страницы

Функциональность и внешний вид графика не меняются. Текст «не обновляется автоматически» заменяется на пояснение, что ИПЦ обновляется из официальной таблицы Росстата. Ссылка ведёт на постоянную страницу раздела, а не на меняющийся XLSX.

## Тестирование

До реализации создаются падающие тесты для следующих сценариев:

1. Разбор минимальной корректной книги Росстата.
2. Выбор самой свежей ссылки и отклонение чужого домена.
3. Добавление одного нового последовательного месяца.
4. Повторный запуск без новых данных не изменяет файлы.
5. Изменение исторического значения вызывает ошибку.
6. Пропуск, дубликат, неверное значение и сломанная структура вызывают ошибку.
7. Генерация сохраняет все даты и значения опубликованного baseline до июня 2025 года.
8. Текущая официальная таблица обновляет ряд до последнего доступного месяца.
9. Сгенерированный `ipc_data.js` синтаксически корректен и создаёт ожидаемый объект `ipc`.

После модульных тестов сайт проверяется локально в браузере: отсутствие ошибок консоли, наличие трёх исходных traces, прежняя первая точка ИПЦ `1.0`, актуальная последняя точка и работа элементов управления. После публикации те же проверки выполняются на публичном URL.

## Критерии готовности

Решение готово, когда:

- ветка синхронизирована с актуальным `origin/master`;
- канонический файл содержит все официальные месяцы до последней публикации Росстата;
- исторический baseline до июня 2025 года проходит побайтовую/числовую проверку;
- локальные тесты и браузерная проверка проходят;
- workflow находится в `master`, запускается вручную и по расписанию;
- последний workflow завершился успешно;
- GitHub Pages опубликован этим workflow;
- публичный сайт показывает актуальный ИПЦ и остаётся работоспособным без запросов к Росстату из браузера;
- исходная незакоммиченная локальная переработка `src/` не изменена.
